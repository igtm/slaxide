## 3. いま詰まっている点

この環境では GTK 系の system package が無く、次が失敗する。

```bash
cargo check -p slaxide-linux-gtk
```

失敗理由:

- `gtk4.pc` が見つからない
- `graphene-gobject-1.0.pc` が見つからない

Arch/Manjaro 系の公式 package 情報では、`gtk4` package が GTK4 のファイルを、`graphene` package が `graphene-gobject-1.0.pc` を提供している。`pkgconf` も必要。

参考:

- https://archlinux.org/packages/extra/x86_64/gtk4/files/
- https://archlinux.org/packages/extra/x86_64/graphene/files/
- https://archlinux.org/packages/core/x86_64/pkgconf/
- https://archlinux.org/packages/extra/x86_64/libadwaita/

## 4. Manjaro 再起動後の最初の作業

まず Manjaro 上で system dependency を入れる。

```bash
sudo pacman -Syu --needed base-devel pkgconf gtk4 graphene libadwaita
```

その後、最低限この順番で確認する。

```bash
pkg-config --modversion gtk4
pkg-config --modversion graphene-gobject-1.0
cargo test
cargo check -p slaxide-linux-gtk
cargo run -p slaxide-linux-gtk
```

Wayland セッション確認:

- GNOME on Wayland で起動確認
- Hyprland で起動確認

### 5.7 通知ルール engine を接続する

## 7. 実装時のチェックポイント

### 7.1 アーキテクチャ

- `slaxide-core` に GTK や reqwest を入れない
- `slaxide-linux-gtk` は薄い frontend に保つ
- macOS 用 frontend を後で足せる crate 境界を壊さない

### 7.2 プロダクト要件

- チャンネル移動を主導線に戻さない
- タイムライン中心を崩さない
- 通知ルールとタイムライン順位付けを混ぜない
- DM を勝手に MVP に入れない
- 1ユーザー / 1ワークスペース前提を崩さない

### 7.3 Linux / Wayland

- Hyprland で DnD / clipboard の挙動を必ず確認する
- GNOME だけで動いて満足しない
- GTK CSS と libadwaita の見た目競合を確認する

### 7.4 Slack API

- internal app 前提を維持する
- external file upload API を使う
- `files.upload` 前提に戻さない
- token rotation を最初から有効にする

## 9. 注意点

- `Cargo.toml` の default-members から `slaxide-linux-gtk` は外してある。理由は system dependency が無い環境でも backend の `cargo test` を回せるようにするため。
- 現在の UI はプロトタイプであり、表示は出る前提だが本番データフローは未接続。
- keyring 実装は `use_native_store(true)` を呼ぶので、Linux では keyutils ではなく Secret Service 寄りの挙動を期待している。セッション側の keyring 解錠状態は確認すること。
- libadwaita を深く使い始めるとテーマ自由度と競合しやすい。Zen テーマ優先なら CSS 適用戦略を早めに見直すこと。
