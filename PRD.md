# Slaxide PRD

更新日: 2026-03-15

## 1. プロダクト概要

Slaxide は、Slack を「チャンネルを移動して読むツール」ではなく、「今見るべき会話に集中するための統合タイムライン」として再構成する Rust 製デスクトップアプリである。

最初の対象は Linux の Manjaro 環境で、特に GNOME / Hyprland / Wayland 上で快適に動作することを重視する。その後に macOS 対応を行う。

## 2. 解決したい課題

- 会社の Slack ではチャンネル数が多く、チャンネル移動自体が認知負荷になる。
- 「未読が多い」ことと「今見るべき」ことは一致しない。
- 返信のためにチャンネルに移動し直す体験が思考を分断する。
- Slack 標準 UI は情報密度が高く、集中を保ちにくい。
- 通知も粒度が荒く、本当に必要なものだけを拾いにくい。

## 3. プロダクトビジョン

- とにかく高速であること。
- Zen スタイルで、視界と操作対象を絞ること。
- 「どこに行くか」ではなく「何に反応するか」に集中できること。
- Twitter の `For You` 的な発想で、Slack の複数チャンネルを横断した優先度付きタイムラインを持つこと。
- 返信はタイムライン上からそのまま行え、チャンネル移動を前提にしないこと。

## 4. 対象ユーザー

- 社内 Slack を日常的に使う個人ユーザー
- チャンネル数が多い会社で、優先度判断コストを減らしたい人
- Linux をメインに使い、将来的には macOS でも同じ思想の体験を欲しい人

初期前提:

- 社内専用アプリ
- 1ユーザー
- 1ワークスペース

## 5. MVP スコープ

### 5.1 対応プラットフォーム

- Phase 1: Linux (Manjaro, GNOME / Hyprland, Wayland)
- Phase 2: macOS

### 5.2 Slack 対象範囲

- 公開チャンネルと非公開チャンネルを対象にする
- DM / MPIM は MVP では対象外
- 直近 7 日の履歴 + ライブ更新を扱う

### 5.3 コア体験

- 統合タイムラインを主画面にする
- 統合タイムラインは「重要 + 監視チャンネル」のハイブリッドで構成する
- メンション、自分が参加しているスレッド、フォーカスキーワード一致、監視対象チャンネルの新着を重み付けして表示する
- `Focus` と `Recent` の 2 モードを持つ
- 右側に thread drawer を持ち、選択中の会話にその場で返信できる
- 返信は Slack スレッド機能を使う
- チャンネル一覧は補助導線であり、主導線はタイムラインである

### 5.4 composer / 返信

- 統合タイムラインから直接返信できる
- テキスト返信をサポートする
- 添付ファイルをサポートする
- Linux Wayland でドラッグ&ドロップをサポートする
- 画像ペーストをサポートする

### 5.5 認証 / 配布

- ブラウザ OAuth を使う
- internal app 前提で設計する
- Linux の最初の deliverable は開発ビルドでよい

## 6. UX 要件

### 6.1 基本原則

- 低認知負荷
- 高速起動
- 余計なサイドバー依存を避ける
- 今見るべきものを太く、その他を細く見せる
- キーボードでもマウスでも快適に操作できる

### 6.2 画面構成

- 左: 最小限のナビゲーション
- 中央: 統合タイムライン
- 右: 選択中スレッドと返信 composer

### 6.3 テーマ

- テーマ切り替え機能を持つ
- モダンなエディタで馴染みのあるテーマを複数内蔵する
- 少なくとも以下の系統を用意する
  - Tokyo Night
  - Catppuccin
  - Kanagawa
  - Nord
  - Gruvbox
  - GitHub Light

## 7. 通知要件

- ネイティブ通知をサポートする
- Linux 上でデスクトップ通知として自然に動作する
- 通知ルールは細かく設定できる
- 通知ルールでは少なくとも以下を指定できる
  - チャンネル
  - 投稿者
  - 文字列一致
  - 正規表現一致
  - 除外条件
  - 自分が参加しているスレッドのみ
  - quiet hours
  - 通知の強さ
- 通知ルールとタイムライン順位付けは別管理にする

## 8. 非機能要件

- Rust を使う
- Linux では Wayland を第一級で扱う
- 将来の macOS 対応のため、UI とドメイン/インフラを分離する
- 資格情報は OS キーチェーン / keyring に保存する
- ローカルキャッシュは SQLite を使う
- メッセージキャッシュは MVP では通常の SQLite 保存でよい

## 9. 既に決めた実装方針

- Linux UI は GTK4 ベースで先行する
- Rust の core / store / slack / platform を UI から分離する
- 将来の macOS では core を再利用し、別 frontend を許容する
- タイムライン順位付けは MVP ではルールベースで実装する
- テーマは MVP では内蔵テーマ群を提供する

## 10. MVP でやらないこと

- 一般公開向け配布・Slack Marketplace 前提の設計
- 複数ワークスペース対応
- 複数アカウント対応
- DM / MPIM 統合
- 学習ベースのランキング
- テーマファイル外部読込互換

## 11. 成功条件

- Linux 上でアプリが起動し、統合タイムラインが主導線として成立する
- タイムラインからチャンネル移動せずにスレッド返信できる
- テーマ切り替えが自然に動く
- 通知フィルターが実用的な粒度で設定できる
- 大量チャンネル環境でも「今見るべきもの」に集中できる体験が出せる
